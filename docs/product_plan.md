# Amazon Inventory Agent Demo — 产品规划

## 1. 项目背景

亚马逊 FBA 卖家日常运营中，库存管理是最核心、最耗时的环节之一。卖家需要同时关注：

- 可售库存是否足够支撑未来销量
- 在途入库是否能在断货前到货
- 哪些 SKU 即将断货、需要紧急补货
- 哪些 SKU 库存积压、存在滞销风险
- 不可售库存、预留库存等异常状态

现实中，卖家往往依赖 Excel 手工汇总、ERP 报表或运营经验做判断，数据分散、更新滞后、规则不统一，难以形成「分析 → 建议 → 任务 → 执行」的闭环。

**Amazon Inventory Agent Demo** 旨在用最小可行产品（MVP）演示：如何将商品、库存、销量、补货配置数据汇聚后，由规则引擎 Agent 自动分析并生成可执行的库存运营任务，再通过 API 与前端完成展示。

> **重要说明**：本项目第一阶段**不是**完整 ERP，而是一个**库存 Agent Demo**。第一阶段只解决库存数据分析和库存任务生成；广告、利润、采购、财务等模块仅作为后续扩展方向在文档中预留。

## 2. 目标用户

| 用户类型 | 描述 |
|---------|------|
| 亚马逊 FBA 卖家 | 管理 50–500+ SKU，需要库存预警与补货建议 |
| 运营主管 | 需要每日待办清单，按优先级处理断货/滞销 |
| 产品经理 / 技术负责人 | 评估 AI Agent 在库存场景的可行性 |
| 内部开发团队 | 基于 Demo 扩展为正式卖家运营系统 |

---

## 3. 用户痛点

| 痛点 | 具体表现 |
|------|---------|
| 数据分散 | 商品、库存、销量、补货参数分布在不同 Excel / 后台页面 |
| 判断依赖经验 | 断货风险、补货量靠人工估算，标准不一致 |
| 缺乏任务闭环 | 发现问题后没有结构化待办，容易遗漏 |
| 无法追溯 | 历史分析依据不清晰，难以复盘决策 |
| 工具过重 | 完整 ERP 实施成本高，小团队用不起 |

## 4. 产品定位

```
┌─────────────────────────────────────────────────────────────┐
│  Amazon Inventory Agent Demo（库存 Agent MVP）              │
├─────────────────────────────────────────────────────────────┤
│  输入：Excel 模拟数据（后续 → Amazon SP-API）                │
│  处理：规则引擎 Agent（不接 LLM）                            │
│  输出：库存分析结果 + 运营待办任务 + API + 前端展示           │
└─────────────────────────────────────────────────────────────┘
```

**一句话定位**：面向亚马逊卖家的库存智能分析 Demo，用规则引擎替代人工判断，生成可执行的库存运营任务。

---

## 5. MVP 核心目标

1. 使用 Excel 模拟 Amazon 商品、库存、销量、补货配置数据
2. 将模拟数据导入 MySQL（6 张核心表）
3. 库存 Agent 第一版使用**规则引擎**，不接 LLM
4. Agent 计算：可售天数、总覆盖天数、有效在途、预计断货日、风险等级、建议补货量、建议动作
5. 根据分析结果自动生成库存运营任务
6. FastAPI 提供 REST 接口
7. React + Vite 前端展示：库存总览、库存列表、今日待办
8. 完整演示「数据接入 → 入库 → Agent 分析 → 建议 → 任务 → API → 前端」闭环

## 8. 核心业务流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────────┐
│  Excel   │───▶│  导入    │───▶│  MySQL   │───▶│  Agent 分析  │
│  模拟数据 │    │  脚本    │    │  6张表   │    │  规则引擎    │
└──────────┘    └──────────┘    └──────────┘    └──────┬───────┘
                                                        │
                       ┌────────────────────────────────┘
                       ▼
              ┌─────────────────┐    ┌──────────────┐
              │ 分析结果写入     │───▶│ 任务生成     │
              │ agent_analysis  │    │ agent_tasks  │
              └────────┬────────┘    └──────┬───────┘
                       │                    │
                       ▼                    ▼
              ┌─────────────────────────────────┐
              │         FastAPI REST API         │
              └─────────────────┬───────────────┘
                                ▼
              ┌─────────────────────────────────┐
              │    React 前端（总览/列表/待办）    │
              └─────────────────────────────────┘
```

### 8.1 日常运营流程（MVP 模拟）

1. **准备数据**：运营填写 / 生成 Excel（商品、库存、销量、补货配置）
2. **导入数据**：运行 `python scripts/import_excel.py`，数据写入 MySQL
3. **触发分析**：调用 API `POST /api/v1/agent/analyze` 或运行分析脚本
4. **查看结果**：前端库存列表查看每个 SKU 的风险与建议
5. **处理待办**：前端今日待办页处理 critical / high 优先级任务
6. **更新状态**：将任务标记为 in_progress / resolved

## 9. 库存 Agent 的业务价值

| 价值点 | 说明 |
|--------|------|
| 标准化判断 | 用统一规则替代个人经验，减少误判 |
| 提前预警 | 在断货前按补货周期给出 critical / high 预警 |
| 滞销识别 | 识别低动销 + 高库存 SKU，避免资金占用 |
| 可执行输出 | 不仅给建议，还生成带优先级的待办任务 |
| 可追溯 | 分析记录关联 snapshot_id / sales_id / config_id |
| 可扩展 | 规则引擎可逐步替换为 LLM Agent，接口不变 |

## 10. 后续扩展方向

### 10.1 数据源扩展

- Amazon SP-API：FBA Inventory API、Reports API、Orders API
- 定时同步库存快照与销量
- Product Fees API（为利润 Agent 准备）

### 10.2 Agent 扩展

| Agent | 职责 |
|-------|------|
| 库存 Agent（当前） | 断货、滞销、补货建议 |
| 利润 Agent | 结合 FBA 费用、广告费计算真实利润 |
| 广告 Agent | ACOS、投放与库存联动 |
| 综合运营 Agent | 多 Agent 结果汇总（需 LLM） |

### 10.3 产品化扩展

- 多店铺 / 多站点
- 审批流与操作日志
- 企业微信 / 邮件告警
- 与 ERP / 采购系统对接
